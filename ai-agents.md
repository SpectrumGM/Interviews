# AI Agents — 10 Core Concepts

## Core Summary

AI Agent — это LLM, которая не просто отвечает на вопросы, а сама решает что делать, вызывает инструменты (поиск, API, код), смотрит на результат, и решает что делать дальше. В отличие от обычного chatbot, агент действует в цикле: думает → действует → наблюдает → думает снова — пока задача не решена.

Агенты — это то, куда движется вся AI-индустрия: от "спроси LLM" к "поручи LLM задачу и она сама разберётся".

---

## Key Concepts

### 1. ReAct Loop (Reason + Act)

Базовый паттерн работы агента — цикл из трёх шагов:

1. **Thought (рассуждение)** — агент думает: "Мне нужно найти цену акции AAPL"
2. **Action (действие)** — вызывает инструмент: `search("AAPL stock price")`
3. **Observation (наблюдение)** — получает результат: "$215.30"
4. Возвращается к шагу 1: "Теперь нужно сравнить с прошлым месяцем..."

Цикл повторяется пока агент не решит что задача выполнена.

**Ключевое отличие от prompt chaining:** в chaining ты заранее задаёшь последовательность шагов. В ReAct агент САМ решает какой шаг следующий на основе того что видит. Это делает агентов гибкими, но и непредсказуемыми.

**Главный риск:** зацикливание — агент повторяет одно и то же действие. Решение: max iterations (лимит шагов), loop detection (если action повторяется 3 раза — стоп), timeout.

---

### 2. Tool Calling / Function Calling

Агент бесполезен без инструментов. Tool calling — механизм, через который LLM вызывает внешние функции.

Как работает: ты описываешь LLM доступные инструменты (название, описание, параметры в JSON Schema). Модель решает какой tool вызвать и с какими параметрами. Результат возвращается модели как observation.

Примеры tools:
- `web_search(query)` — поиск в интернете
- `run_code(python_code)` — выполнение кода
- `send_email(to, subject, body)` — отправка письма
- `query_database(sql)` — запрос к базе данных
- `get_weather(city)` — получить погоду

**Описание tool критично:** если описание нечёткое, модель будет вызывать неправильные инструменты или с неправильными параметрами. Это как давать человеку набор инструментов без инструкции — он будет пытаться забивать шуруп молотком.

**Parallel tool calling** — продвинутые модели (GPT-4, Claude) могут вызывать несколько tools одновременно, если они независимы. Экономит время.

### 3. Planning & Self-Correction

Сложные задачи требуют плана. Агент может:

**Task decomposition** — разбить большую задачу на подзадачи:
> "Подготовь отчёт о конкурентах" →
> 1. Найти список конкурентов
> 2. Для каждого: найти выручку, продукты, последние новости
> 3. Сравнить метрики
> 4. Написать summary

**Reflection** — после выполнения шага агент оценивает свой результат: "Это хороший ответ? Я что-то пропустил? Нужно ли переделать?"

**Replanning** — если план не работает (tool вернул ошибку, данные неполные), агент перестраивает план на ходу.

**Trade-off:** planning улучшает качество, но добавляет latency (каждый шаг размышления = дополнительные API calls и токены). Для простых задач planning — overkill. Для сложных — необходимость.

---

### 4. Memory Systems (Short/Long-term)

LLM по умолчанию не помнит ничего между вызовами. Memory systems добавляют "память":

**Short-term memory (контекст разговора)** — история сообщений в текущей сессии. Ограничена context window модели (128k токенов у GPT-4, 200k у Claude). При длинных разговорах старые сообщения обрезаются или суммаризируются.

**Conversation summarization** — вместо хранения всех сообщений, LLM периодически сжимает историю в краткое summary. Экономит токены, но теряет детали.

**Long-term memory (vector memory)** — важные факты из разговоров сохраняются в vector DB. При новом запросе агент ищет релевантные воспоминания и подкидывает в контекст. Пример: "Пользователь предпочитает краткие ответы" — сохраняется и используется в будущих сессиях.

**Working memory** — промежуточные результаты текущей задачи. Агент выполняет шаг 1, сохраняет результат, использует его в шаге 2. Без этого агент "забывает" что нашёл на предыдущих шагах.

**Context window explosion** — главная проблема: инструкции + tools описание + history + memory + retrieved docs + текущий запрос — всё должно влезть в контекст. При сложных задачах это быстро упирается в лимит.

### 5. Multi-Agent Collaboration

Вместо одного агента — команда из нескольких, каждый со своей ролью:

**Supervisor (менеджер)** — получает задачу, решает какому worker-у её отдать, координирует.

**Worker (исполнитель)** — специализированный агент с конкретными tools и системным промптом. Один worker для кода, другой для поиска, третий для анализа данных.

**Critic (проверяющий)** — отдельный агент, который проверяет результат worker-а на ошибки и качество.

Frameworks для multi-agent:
- **LangGraph** — от создателей LangChain. Агенты как узлы графа, messages передаются по рёбрам. Самый гибкий.
- **CrewAI** — роли (role), цели (goal), backstory для каждого агента. Проще начать, но менее гибкий.
- **AutoGen (Microsoft)** — агенты общаются друг с другом в "чате". Хорош для задач где нужна дискуссия.

**Coordination overhead** — главная проблема. Агенты тратят токены на общение между собой. Плохо спроектированная мульти-агентная система может стоить в 5-10x дороже одного хорошего агента при том же качестве.

### 6. Guardrails & Safety Layers

Агент может выполнять реальные действия (отправить email, удалить файл, потратить деньги). Без guardrails это опасно.

**Input validation** — проверка входящего запроса: нет ли prompt injection (пользователь пытается перехватить контроль над агентом)?

**Output validation** — проверка ответа агента перед выполнением: "Ты точно хочешь отправить этот email?" / "Этот SQL запрос удаляет все данные — точно выполнить?"

**Action confirmation** — для опасных действий (отправка, удаление, оплата) требовать подтверждение от пользователя.

**Fact-checking** — перед ответом пользователю отдельная модель проверяет: утверждения основаны на фактах или галлюцинации?

**Scope limiting** — агент может вызывать ТОЛЬКО разрешённые tools. Нельзя давать агенту доступ к `execute_shell_command()` в production без жёстких ограничений.

Инструменты: NVIDIA NeMo Guardrails, LlamaGuard, Guardrails AI.

### 7. Evaluation of Agents

Оценка агентов сложнее чем оценка обычных LLM — нужно мерить не только качество ответа, но и процесс:

**Success rate** — процент задач, которые агент выполнил правильно. Звучит просто, но определить "правильно" для open-ended задач сложно.

**Cost per task** — сколько токенов / API calls / денег ушло на одну задачу. Агент может решить задачу, но потратить $5 на то, что можно было за $0.10.

**Loop detection** — как часто агент зацикливается. Высокий loop rate = плохой planning или неправильные tool descriptions.

**Steps to completion** — сколько шагов нужно агенту для решения. Меньше шагов (при том же качестве) = лучше.

**Tool accuracy** — вызывает ли агент правильные tools с правильными параметрами?

**Latency** — время от запроса до ответа. Агенты медленнее обычных LLM (каждый шаг ReAct = отдельный API call).

Автоматизация eval сложна: можно создать тестовые сценарии с expected outcomes, но реальные задачи всегда разнообразнее тестов.

### 8. MCP Servers (Model Context Protocol)

**MCP (Model Context Protocol)** — стандартизированный протокол для подключения инструментов к LLM. Разработан Anthropic.

Проблема без MCP: каждый tool нужно описывать вручную для каждой модели. Подключение к Slack, Gmail, GitHub — каждый раз писать свою интеграцию.

MCP решает это как USB-C: один стандартный интерфейс для любого инструмента и любой модели.

**MCP Server** — сервис, который предоставляет tools через стандартный протокол. Пример: MCP server для GitHub предоставляет tools: `create_issue()`, `search_repos()`, `create_pr()`. Любая LLM, поддерживающая MCP, может вызывать эти tools без кастомного кода.

**MCP Client** — LLM приложение (Claude, IDE, agent framework), которое подключается к MCP серверам и использует их tools.

Преимущества:
- Один раз написал MCP server — работает с любым клиентом
- Security: стандартизированная авторизация и scope ограничения
- Ecosystem: растущая библиотека готовых MCP серверов

Протокол ещё молодой (2024-2025), но быстро становится стандартом. Claude Desktop, Claude Code, Cursor, Windsurf уже поддерживают MCP.

### 9. Streaming & Async Execution

**Streaming** — отправка ответа пользователю по мере генерации, token за token, а не целиком в конце. Критически важно для UX: пользователь видит что агент "думает" и работает, а не ждёт 30 секунд в тишине.

**Async tool execution** — агент вызывает tool, но не блокирует выполнение пока ждёт результат. Может параллельно вызвать другие tools или начать генерировать промежуточный ответ.

Пример: агент ищет в трёх базах данных одновременно (async), показывает "Searching..." пользователю (streaming), и начинает генерировать ответ по мере получения результатов.

**Complex state management** — главная сложность. При async execution нужно отслеживать: какие tools запущены, какие вернули результат, какие зафейлились, в каком порядке обрабатывать результаты. Race conditions, error handling, retry logic — всё как в обычном async programming, но с добавлением непредсказуемости LLM.

Frameworks: LangGraph поддерживает streaming и async из коробки. Vercel AI SDK — для frontend streaming.

### 10. Cost & Latency Optimization

Агенты дорогие: каждый шаг ReAct = API call (токены + latency). 10-шаговый агент может стоить 10x от одного LLM запроса.

**Model routing** — не все шаги требуют сильную модель. Planning → Claude/GPT-4 (нужно рассуждение). Simple tool calls → GPT-4o-mini/Haiku (дешевле и быстрее). Роутер решает какую модель использовать для каждого шага.

**Caching** — если агент часто делает одинаковые tool calls (одни и те же поисковые запросы), кэшировать результаты. Prompt caching — если начало промпта одинаковое (system prompt + tools), модели могут кэшировать prefix.

**Early stopping** — если агент уже нашёл ответ, не нужно продолжать "для надёжности". Детектировать когда задача решена и останавливаться.

**Tool routing** — если есть несколько tools для одной задачи (дорогой точный vs дешёвый приближённый), выбирать в зависимости от сложности запроса.

**Batching** — группировать несколько независимых tool calls в один batch вместо sequential execution.

На масштабе (тысячи запросов в день) оптимизация может снизить стоимость в 3-5x без потери качества.

## Architecture Flow

**User Task** ("Подготовь отчёт о конкурентах")

⬇️ **Planning** — агент разбивает задачу на подзадачи

⬇️ **ReAct Loop начинается:**

⬇️ **Thought** — "Сначала найду список конкурентов"

⬇️ **Action** — вызов tool: `web_search("competitors of [company]")`

⬇️ **Observation** — результат поиска

⬇️ **Thought** — "Нашёл 5 конкурентов. Теперь нужна выручка каждого"

⬇️ **Action** — вызов tool: `query_database("SELECT revenue FROM...")`

⬇️ **Observation** — данные из базы

⬇️ ... (цикл продолжается)

⬇️ **Self-Correction** — "Результат неполный, нужно поискать ещё"

⬇️ **Guardrails Check** — валидация финального ответа

⬇️ **Response** — готовый отчёт пользователю

## Examples

**Пример 1 — Research agent:**
- User: "Найди топ-5 AI стартапов в healthcare, их funding и ключевые продукты"
- Planning: 1) search стартапы 2) для каждого: найти funding 3) найти продукты 4) составить таблицу
- Tools: web_search, scrape_website
- Multi-query: параллельный поиск по каждому стартапу (async)
- Self-correction: если funding не найден — попробовать другой источник (Crunchbase)
- Output: structured markdown таблица

**Пример 2 — DevOps agent:**
- User: "CI pipeline сломался, разберись"
- Tools: `read_logs()`, `query_github()`, `run_tests()`, `create_pr()`
- ReAct: читает логи → определяет ошибку → находит problematic commit → пишет fix → создаёт PR
- Guardrails: PR требует human approval перед merge
- Memory: запоминает частые ошибки для быстрого resolution в будущем

## Why This Matters

AI Agents — следующий шаг после chatbots. Chatbot отвечает на вопросы. Agent выполняет задачи. Это фундаментальный сдвиг: от "AI как консультант" к "AI как исполнитель".

Компании уже строят агентов для customer support, код-ревью, data analysis, sales outreach. Через 2-3 года большинство рутинных бизнес-процессов будут частично или полностью выполняться агентами.

Для инженера: понимание agent architecture (ReAct, tools, memory, multi-agent, guardrails) — это то, что отличает AI engineer от обычного developer, который просто вызывает API.

## Glossary

| Term | Что это |
|------|---------|
| ReAct | Reason + Act — паттерн где агент чередует рассуждение и действие в цикле |
| Tool calling | Механизм вызова внешних функций (API, поиск, код) из LLM |
| Function calling | Другое название tool calling — модель генерирует JSON с именем функции и параметрами |
| Task decomposition | Разбиение сложной задачи на подзадачи |
| Reflection | Агент оценивает свой собственный результат и решает нужно ли переделать |
| Context window | Максимальный объём текста, который модель может видеть одновременно (в токенах) |
| Loop detection | Обнаружение зацикливания агента (повторяет одни и те же действия) |
| MCP | Model Context Protocol — стандарт подключения инструментов к LLM |
| Streaming | Отправка ответа пользователю по мере генерации, а не целиком в конце |
| Async execution | Параллельное выполнение нескольких tool calls, не дожидаясь каждого по очереди |
| Model routing | Выбор разных моделей для разных шагов (сильная для planning, дешёвая для простого) |
| Early stopping | Остановка агента как только задача решена, без лишних шагов |
| Guardrails | Защитные слои: валидация input/output, подтверждение опасных действий |
| Cross-encoder | Модель оценивающая пару (запрос, документ) вместе — точнее но медленнее |
| Prompt caching | Кэширование начала промпта (system prompt + tools) для ускорения повторных вызовов |
