# RAG (Retrieval-Augmented Generation) — 10 Core Concepts


## Core Summary

RAG — это когда LLM перед ответом сначала ищет релевантные документы в базе знаний и использует их как контекст. Вместо того чтобы полагаться только на то что модель запомнила при обучении, ты подкидываешь ей актуальные данные прямо в промпт. Это решает главные проблемы LLM: устаревшие знания, галлюцинации, отсутствие domain-specific информации.

Pipeline: документы → нарезка на чанки → embedding → хранение в vector DB → поиск по запросу → контекст в LLM → ответ.

---

## Key Concepts

### 1. Retrieval Pipeline (общая архитектура)

Базовый RAG pipeline состоит из двух фаз:

**Ingestion (offline)** — подготовка базы знаний:
1. Загрузка документов (PDF, HTML, Markdown, базы данных)
2. Chunking — нарезка на куски
3. Embedding — каждый чанк превращается в вектор
4. Сохранение в vector DB с метаданными

**Retrieval + Generation (online)** — ответ на запрос:
1. Запрос пользователя → embedding
2. Поиск ближайших чанков в vector DB
3. (опционально) Reranking результатов
4. Найденные чанки вставляются в промпт как контекст
5. LLM генерирует ответ на основе контекста

---

### 2. Chunking Strategies

Chunking — нарезка документов на куски перед embedding. Размер и способ нарезки критически влияют на качество поиска.

**Fixed-size chunking** — режем по количеству символов/токенов (например, по 512 токенов с overlap 50). Просто, но может резать предложения и абзацы пополам.

**Semantic chunking** — режем по смысловым границам (абзацы, секции, смена темы). Embedding модель определяет где меняется тема. Лучше сохраняет контекст, но сложнее реализовать.

**Hierarchical chunking** — несколько уровней: большие чанки (целые секции) для общего контекста + маленькие чанки (абзацы) для точного поиска. При поиске находим маленький чанк, но в контекст LLM отдаём его parent (большой чанк).

**Overlap** — перекрытие между соседними чанками (обычно 10-20% размера). Чтобы информация на границе чанков не терялась.

Слишком маленькие чанки — потеря контекста. Слишком большие — шум и лишние токены. Типичный sweet spot: 256-1024 токена.

---

### 3. Vector Embeddings & Similarity Search

**Embedding** — превращение текста в вектор (массив чисел), кодирующий смысл. Похожие по смыслу тексты = близкие векторы.

**Dense embeddings** — стандартные embedding модели (OpenAI text-embedding-3, BGE, E5). Каждый элемент вектора — число с плавающей запятой. Ловят семантику, но могут пропускать exact matches.

**Sparse embeddings** — вектор в основном из нулей, ненулевые значения соответствуют конкретным словам/токенам. По сути продвинутый keyword matching. BM25 — классический sparse метод. Хорош для точных терминов, кодов, имён.

**Hybrid search** — комбинация dense + sparse. Ловит и смысл, и точные совпадения. Почти всегда лучше чем только один тип.

Выбор embedding модели критичен: плохой embedding = нерелевантный retrieval = LLM галлюцинирует даже при правильной архитектуре.

---

### 4. Vector Databases

Специализированные базы данных для хранения и быстрого поиска по векторам:

| База | Тип | Когда использовать |
|------|-----|-------------------|
| Pinecone | Managed (облако) | Быстрый старт, не хочешь думать про инфру |
| Weaviate | Self-hosted / Cloud | Нужен GraphQL API и встроенные embedding модули |
| Chroma | Self-hosted | Прототипы, локальная разработка, маленькие проекты |
| PGVector | Расширение PostgreSQL | Уже есть Postgres, не хочешь отдельную базу |

**Freshness management** — главная операционная проблема vector DB. Документы обновляются, но их старые embeddings остаются в базе. Нужен pipeline для re-embedding и upsert при изменениях.

### 5. Advanced RAG Patterns

Базовый RAG (retrieve top-K чанков → вставить в промпт) работает, но имеет потолок качества. Advanced patterns:

**Parent-document retrieval** — ищем по маленьким чанкам (точный match), но в контекст LLM отдаём целый parent-документ или секцию. Точность поиска + полнота контекста.

**Hypothetical Document Embedding (HyDE)** — вместо поиска по запросу пользователя, сначала просим LLM сгенерировать гипотетический ответ, и ЭТОТ ответ используем для поиска. Идея: сгенерированный ответ ближе по embedding-пространству к реальным документам, чем короткий вопрос.

**Reranking** — после vector search пропускаем результаты через отдельную модель-реранкер (Cohere Rerank, BGE Reranker). Реранкер точнее оценивает релевантность, потому что видит и запрос, и документ вместе (cross-encoder), в отличие от embedding (bi-encoder, кодирует их отдельно).

**Multi-query** — один запрос пользователя переформулируется в несколько вариантов, каждый ищется отдельно, результаты объединяются. Ловит разные аспекты вопроса.

---

### 6. Evaluation Metrics (RAGAS)

RAGAS — фреймворк для автоматической оценки RAG систем. Ключевые метрики:

**Faithfulness** — насколько ответ LLM соответствует найденным документам (а не выдумке модели). Высокий faithfulness = модель не галлюцинирует, а отвечает строго по контексту.

**Answer relevance** — насколько ответ релевантен вопросу пользователя. Модель может точно цитировать документ, но не отвечать на заданный вопрос.

**Context recall** — какой процент нужной информации был найден retrieval-ом. Если retrieval пропустил важный документ — LLM не сможет дать полный ответ.

**Context precision** — какой процент найденных документов реально нужен. Много нерелевантного контекста = шум, путает LLM.

Эти метрики считаются автоматически (LLM-as-judge), но не идеально — нужна калибровка и периодическая проверка human eval.

### 7. Hallucination Mitigation in RAG

RAG снижает галлюцинации, но не убирает их полностью. LLM всё ещё может выдумывать или неправильно интерпретировать контекст.

Методы борьбы:

**Citation / attribution** — заставить модель указывать источник каждого утверждения. Prompt: "Cite the document chunk for every claim. If no source, say 'I don't know'."

**Grounding checks** — отдельная модель или pipeline проверяет: каждое утверждение в ответе подтверждено найденными документами? Если нет — отмечает как ungrounded.

**Constrained generation** — ограничить модель: отвечать ТОЛЬКО на основе предоставленного контекста, не использовать внутренние знания.

**Confidence scoring** — модель или отдельный классификатор оценивает уверенность в ответе. Низкая уверенность → эскалация на человека.

Для enterprise — это must have. Один галлюцинированный юридический совет или медицинская рекомендация = серьёзные последствия.

### 8. Multi-Document & Multi-Turn RAG

**Multi-document** — когда ответ требует информации из нескольких документов. Пример: "Сравни условия договора A и договора B." Нужно найти чанки из обоих документов и синтезировать ответ.

Проблема: стандартный top-K retrieval может вернуть чанки только из одного документа. Решение: metadata filtering (искать по каждому документу отдельно) или multi-query.

**Multi-turn** — чат с историей. Пользователь задаёт follow-up вопросы, которые ссылаются на предыдущие ответы.

Пример:
> User: "Какие условия расторжения в договоре?"
> Assistant: "По пункту 5.2, расторжение возможно с уведомлением за 30 дней..."
> User: "А какие штрафы за это?"

"За это" ссылается на расторжение из предыдущего сообщения. Для retrieval нужно переформулировать запрос с учётом истории: "штрафы за расторжение договора" вместо "штрафы за это".

**Query rewriting** — LLM переписывает запрос пользователя, подставляя контекст из истории, перед отправкой в retrieval. Без этого поиск по "штрафы за это" вернёт мусор.

**Token limits** — история + найденные документы + system prompt должны влезать в контекст модели. При длинных разговорах нужно сжимать или отбрасывать старую историю.

---

### 9. Hybrid Search & Reranking

Подробнее о комбинации методов поиска:

**Keyword search (BM25)** — считает релевантность по частоте слов. "Error code 4032" → точно найдёт документ с этим кодом. Но "как исправить проблему с подключением" может не найти документ "troubleshooting connectivity issues".

**Vector search** — ловит семантику. "Как исправить проблему с подключением" → найдёт "troubleshooting connectivity issues". Но может пропустить exact match по коду ошибки.

**Hybrid** — запускаем оба поиска параллельно, объединяем результаты. Методы объединения:
- **Reciprocal Rank Fusion (RRF)** — объединяет ранжирования из разных поисков. Документ высоко в обоих поисках = высоко в итоговом.
- **Weighted combination** — вручную задаёшь веса (например 0.7 × vector + 0.3 × keyword).

**Reranker** — после объединения, отдельная модель пересматривает top-20-50 результатов и выбирает top-5 самых релевантных. Cross-encoder реранкер видит запрос и документ вместе, поэтому точнее bi-encoder embedding.

Добавление reranker обычно даёт +5-15% к качеству retrieval при добавлении 50-200ms latency.

### 10. Production RAG Pipelines

Что нужно чтобы RAG работал в production 24/7:

**Caching** — кэширование embedding запросов и результатов поиска. Одинаковый запрос не нужно embedding + search заново. Semantic cache — кэширование не только exact matches, но и семантически похожих запросов.

**Monitoring** — отслеживание:
- Retrieval quality: средний recall, процент запросов без результатов
- Latency: embedding time + search time + reranking time + LLM time
- Token usage и стоимость
- User feedback (thumbs up/down)

**Drift detection** — со временем данные и запросы пользователей меняются. Embedding модель может становиться менее точной. Нужно отслеживать деградацию метрик и периодически пересчитывать embeddings или менять модель.

**Fallback strategy** — если retrieval вернул 0 результатов или все с низким score, не пытаться отвечать из ничего. Варианты: "Я не нашёл информацию по этому вопросу", escalation на человека, fallback на general LLM knowledge с disclaimer.

## Architecture Flow

**Documents** (PDF, HTML, Markdown, DB)

⬇️ **Chunking** — нарезка на куски (fixed / semantic / hierarchical)

⬇️ **Embedding** — каждый чанк → вектор (OpenAI, BGE, E5)

⬇️ **Vector DB** — сохранение векторов + метаданные (Pinecone, Qdrant, Chroma)

--- ingestion выше, retrieval ниже ---

**User Query**

⬇️ **Query Rewriting** — переформулировка с учётом истории чата (для multi-turn)

⬇️ **Embedding** — запрос → вектор

⬇️ **Hybrid Search** — vector search + BM25 keyword search параллельно

⬇️ **Reranking** — cross-encoder пересортировывает top результаты

⬇️ **Context Assembly** — top-K чанков + system prompt + chat history → промпт

⬇️ **LLM Generation** — модель отвечает на основе контекста

⬇️ **Grounding Check** — проверка что ответ основан на найденных документах

## Examples

**Пример 1 — Customer support RAG:**
- 5000 статей knowledge base → semantic chunking (по секциям) → embed через BGE-large → Qdrant
- Hybrid search + Cohere Rerank → top-5 чанков в промпт
- Grounding check: каждый claim в ответе должен иметь citation
- Fallback: если confidence < 0.6 → "Let me connect you with a human agent"
- Monitoring: weekly RAGAS eval на 200 реальных запросов

**Пример 2 — Legal document Q&A:**
- Multi-document: пользователь загружает несколько контрактов
- Hierarchical chunking: мелкие чанки для поиска, parent sections для контекста
- Multi-turn: follow-up вопросы про конкретные пункты договора
- Query rewriting: "А что насчёт штрафов?" → "Штрафные санкции по пункту 7 договора аренды"
- Citation обязателен: каждое утверждение ссылается на конкретный пункт документа

## Why This Matters

RAG — самый распространённый паттерн в production LLM приложениях. Это первое что строят когда нужно чтобы LLM работала с конкретными данными компании. Без RAG модель ограничена только тем что запомнила при обучении — это не подходит для актуальных, приватных или domain-specific данных.

Качество RAG системы определяется retrieval-ом: если нашли правильные документы — LLM почти наверняка даст правильный ответ. Если retrieval плохой — даже GPT-4 будет галлюцинировать. Поэтому chunking, embedding, hybrid search и reranking — это не мелочи, а core architecture decisions.

## Glossary

| Term | Что это |
|------|---------|
| RAG | Retrieval-Augmented Generation — LLM ищет документы перед ответом |
| Chunking | Нарезка документов на куски для embedding и поиска |
| Embedding | Вектор (массив чисел), кодирующий смысл текста |
| Dense embedding | Стандартный embedding — все элементы значимые, ловит семантику |
| Sparse embedding | Вектор из почти всех нулей, ловит exact keyword matches |
| Hybrid search | Комбинация vector search + keyword search |
| BM25 | Классический алгоритм текстового поиска по частоте слов |
| Reranking | Пересортировка результатов поиска более точной моделью (cross-encoder) |
| Cross-encoder | Модель, которая видит запрос и документ вместе — точнее но медленнее |
| Bi-encoder | Модель, которая кодирует запрос и документ отдельно — быстрее но менее точно |
| HyDE | Hypothetical Document Embedding — поиск по сгенерированному гипотетическому ответу |
| Faithfulness | Метрика: насколько ответ LLM основан на найденных документах (а не выдумке) |
| Context recall | Метрика: какой процент нужной информации retrieval нашёл |
| Grounding | Привязка ответа модели к конкретным документам-источникам |
| Query rewriting | Переформулировка запроса с учётом истории чата для лучшего поиска |
| Semantic cache | Кэширование не только exact matches, но и семантически похожих запросов |
| RRF | Reciprocal Rank Fusion — метод объединения результатов из разных поисков |
| Ingestion | Процесс загрузки, обработки и индексации документов в vector DB |
