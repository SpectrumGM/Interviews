# Vector Databases & Embeddings — 10 Core Concepts

> Source: [@ConsciousRide](https://x.com/ConsciousRide/status/2048064390642200744) · Apr 25, 2026

## Core Summary

Vector databases — это хранилища, заточенные под поиск по смыслу, а не по ключевым словам. Текст (или картинку, аудио) превращают в вектор — массив чисел, который отражает "смысл". Похожие по смыслу тексты = близкие векторы в пространстве. Vector DB умеет быстро находить ближайшие векторы среди миллиардов. Это основа RAG (когда LLM ищет нужные документы перед ответом), рекомендательных систем и семантического поиска.

---

## Key Concepts

### 1. Embeddings & Vector Search

**Embedding** — превращение текста (или любых данных) в вектор фиксированной длины. Например, предложение "Как варить борщ" превращается в массив из 768 или 1536 чисел. Эти числа кодируют смысл — похожие предложения получают похожие векторы.

**Vector search** — поиск ближайших векторов к запросу. Расстояние между векторами = мера похожести (cosine similarity, dot product, euclidean distance).

Модели для embedding: OpenAI text-embedding-3, Cohere embed, open-source варианты (BGE, E5, GTE). Выбор модели напрямую влияет на качество поиска.

**Проблема "curse of dimensionality"** — чем больше размерность вектора (768, 1536, 4096), тем сложнее и дороже поиск. В высоких размерностях все точки становятся "примерно одинаково далеко" друг от друга, что снижает качество.

---

### 2. ANN Algorithms (HNSW, IVF, PQ)

**Exact nearest neighbor** — проверить каждый вектор в базе. Точно, но при миллионах векторов — невозможно медленно.

**ANN (Approximate Nearest Neighbors)** — алгоритмы, которые находят "примерно ближайших" соседей за доли секунды, жертвуя крошечной точностью.

Основные алгоритмы:
- **HNSW (Hierarchical Navigable Small World)** — граф из нескольких уровней. Поиск начинается с верхнего "грубого" уровня и спускается к точному. Самый популярный, хорошее качество, но жрёт RAM.
- **IVF (Inverted File Index)** — делит всё пространство на кластеры. При поиске проверяет только ближайшие кластеры, а не все векторы. Быстрый, но нужен этап обучения (кластеризация).
- **PQ (Product Quantization)** — сжимает векторы, разбивая на подвекторы и квантизируя каждый. Экономит память в 10-50x, но теряет точность.

Часто комбинируют: IVF+PQ или HNSW+PQ для баланса скорости, памяти и точности.

### 3. Popular Vector DBs

| База | Тип | Особенность |
|------|-----|-------------|
| Pinecone | Managed (облако) | Проще всего начать, платный, serverless |
| Weaviate | Self-hosted / Cloud | GraphQL API, встроенные модули для embedding |
| Chroma | Self-hosted | Легковесный, популярен для прототипов и локальной разработки |
| PGVector | Расширение PostgreSQL | Если уже используешь Postgres — не нужна отдельная база |
| Milvus | Self-hosted / Cloud (Zilliz) | Масштабируется до миллиардов векторов |
| Qdrant | Self-hosted / Cloud | Rust-based, быстрый, хорошая фильтрация по метаданным |

**Managed** (Pinecone, Zilliz Cloud) — не думаешь про инфраструктуру, но vendor lock-in и дороже. **Self-hosted** (Chroma, Qdrant, Milvus) — полный контроль, но сам управляешь серверами.

### 4. Indexing Strategies

**Index** — структура данных, которая определяет КАК база ищет ближайших соседей. Выбор индекса = главный trade-off между скоростью, памятью и точностью.

- **Flat (brute force)** — никакого индекса, проверяет все векторы. 100% точность, но не масштабируется. Ок для менее 100k векторов.
- **HNSW** — быстрый поиск, но весь индекс должен влезать в RAM. Для 1M векторов x 768 dims = ~3GB RAM.
- **DiskANN** — хранит индекс на SSD, не в RAM. Позволяет работать с миллиардами векторов на обычных машинах.
- **GPU indexing** — ускоряет построение и поиск на GPU (FAISS GPU). Отлично для batch-запросов.

Построение индекса занимает время (минуты-часы для миллионов векторов). При добавлении новых данных индекс нужно обновлять или перестраивать.

---

### 5. Hybrid Search

Чисто векторный поиск ловит смысл, но может пропустить точные совпадения. Например, запрос "error code 4032" по смыслу похож на "ошибка 4031", но тебе нужен именно 4032.

**Hybrid search** — комбинация:
1. **Vector search** — находит семантически похожие документы
2. **Keyword search (BM25)** — классический текстовый поиск по точным словам
3. **Reranking** — отдельная модель (например Cohere Rerank, BGE Reranker) пересортировывает результаты обоих поисков по релевантности

Результат: лучший recall (находит больше нужного) И лучший precision (меньше мусора). Почти все production RAG системы используют hybrid.

---

### 6. Filtering & Metadata

Каждый вектор в базе можно хранить с метаданными — дополнительными полями (дата, категория, автор, язык, source).

**Pre-filtering** — сначала фильтруешь по метаданным, потом ищешь среди отфильтрованных. Пример: "найди похожие документы, но ТОЛЬКО из категории Finance и ТОЛЬКО за 2025 год".

**Post-filtering** — сначала векторный поиск, потом отбрасываешь результаты не подходящие по фильтрам. Проще, но может вернуть мало результатов.

Pre-filtering эффективнее, но требует специальных индексов по метаданным. Не все базы делают это одинаково хорошо — Qdrant и Weaviate сильны в этом.

---

### 7. Upsert, Deletion & Freshness

**Upsert** — вставить новый вектор или обновить существующий (если ID совпадает). Важно для баз знаний, где документы меняются.

**Deletion** — удаление устаревших векторов. Без удаления модель может находить и цитировать outdated информацию.

**Freshness pipeline** — автоматический процесс: документ обновился → пересчитать embedding → upsert в базу → удалить старый вектор.

Проблема: **re-indexing cost**. Каждый upsert может требовать перестройки части индекса. При большом объёме обновлений это становится дорого. Batch upserts и incremental indexing помогают.

---

### 8. Scalability & Sharding

Когда векторов становится миллиарды, одна машина не справляется.

**Sharding** — разделение индекса на куски (шарды), каждый на своём сервере. Запрос идёт параллельно во все шарды, результаты объединяются.

**Replication** — копии шардов для отказоустойчивости и распределения нагрузки.

Milvus и Qdrant поддерживают шардирование из коробки. Pinecone делает это автоматически (managed). PGVector плохо масштабируется горизонтально — это его слабое место.

**Consistency challenge** — при шардировании сложно гарантировать что все шарды имеют одинаковые данные в один момент времени (eventual consistency vs strong consistency).

---

### 9. Evaluation Metrics

Как понять что твой vector search работает хорошо:

- **Recall@K** — из всех релевантных документов, какой процент попал в top-K результатов? Recall@10 = 0.95 значит 95% нужных документов попадают в top-10.
- **Precision@K** — какой процент результатов в top-K действительно релевантен? Если из 10 результатов 7 полезны — Precision@10 = 0.7.
- **NDCG (Normalized Discounted Cumulative Gain)** — учитывает не только попадание в top-K, но и позицию. Релевантный документ на 1-м месте ценнее чем на 10-м.
- **Latency** — время ответа. Для production обычно цель: p99 < 100ms (99% запросов отвечают быстрее 100 миллисекунд).

Метрики зависят от use case: для RAG важнее recall (не пропустить нужный документ), для рекомендаций — precision (не показать мусор).

---

### 10. Production Considerations

Что нужно помимо самого vector search:

- **Cost** — managed DB (Pinecone) стоит $70-700+/month в зависимости от объёма. Self-hosted дешевле, но нужен DevOps.
- **Monitoring** — отслеживать latency, throughput, index size, embedding drift (качество embeddings деградирует если данные меняются, а модель нет).
- **Backup** — потеря индекса = потеря всех embeddings. Пересчитывать с нуля дорого (API calls к embedding модели).
- **Security** — данные в векторах можно частично восстановить (embedding inversion attacks). Для sensitive data нужна encryption at rest и access control.

## Architecture Flow

**User Query** (текстовый запрос)

⬇️ **Embedding Model** — превращает запрос в вектор (768/1536 dims)

⬇️ **Pre-filtering** — фильтрация по метаданным (дата, категория, source)

⬇️ **ANN Search** — HNSW/IVF/PQ находит top-K ближайших векторов

⬇️ **Keyword Search (BM25)** — параллельный текстовый поиск (hybrid)

⬇️ **Merge + Rerank** — объединение результатов, пересортировка моделью-реранкером

⬇️ **Return top results** — документы уходят в LLM как контекст (RAG) или пользователю напрямую

## Examples

**Пример 1 — RAG для внутренней документации компании:**
- 10 000 документов → chunk по 512 токенов → embed через OpenAI text-embedding-3-small → Qdrant
- Hybrid search: vector + BM25 → Cohere Rerank → top-5 чанков в контекст LLM
- Metadata filtering: отдел, дата обновления, уровень доступа
- Freshness: webhook при обновлении документа → re-embed → upsert

**Пример 2 — Семантический поиск по товарам (e-commerce):**
- 1M товаров → embed название + описание через BGE-large → Milvus
- Pre-filtering: категория, ценовой диапазон, наличие на складе
- GPU indexing для быстрой перестройки при добавлении товаров
- Evaluation: Recall@20 = 0.92, p99 latency < 80ms

## Why This Matters

Vector databases — инфраструктурный слой, на котором стоит весь RAG и семантический поиск. Без них LLM ограничена только тем что помнит из pretraining. С vector DB модель может "смотреть" в актуальные документы, базы знаний, логи — и отвечать точно.

Выбор vector DB, алгоритма индексации и стратегии hybrid search напрямую определяет качество RAG системы. Плохой retrieval = LLM получает нерелевантные документы = галлюцинации и плохие ответы, даже если модель сама по себе сильная.

## Glossary

| Term | Что это |
|------|---------|
| Embedding | Вектор (массив чисел), кодирующий смысл текста/изображения/аудио |
| Vector | Массив чисел фиксированной длины, точка в многомерном пространстве |
| Cosine similarity | Мера похожести двух векторов по углу между ними. 1 = идентичны, 0 = не связаны |
| ANN | Approximate Nearest Neighbors — быстрый приближённый поиск ближайших векторов |
| HNSW | Алгоритм ANN на основе многоуровневого графа — быстрый, но требует RAM |
| IVF | Алгоритм ANN на основе кластеризации — делит пространство на регионы |
| PQ | Product Quantization — сжатие векторов для экономии памяти |
| BM25 | Классический алгоритм текстового поиска по ключевым словам (не семантический) |
| Reranking | Пересортировка результатов поиска отдельной моделью для улучшения точности |
| Sharding | Разделение базы на части (шарды) по разным серверам для масштабирования |
| Upsert | Вставить или обновить запись (insert + update) |
| Recall@K | Метрика: какой процент релевантных документов попал в top-K результатов |
| NDCG | Метрика качества ранжирования, учитывает позицию релевантных результатов |
| Embedding drift | Деградация качества embeddings когда данные меняются, а модель остаётся старой |
| p99 latency | Время ответа, в которое укладываются 99% запросов |
