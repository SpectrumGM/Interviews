# Fine-Tuning for LLM Systems — 9 Core Techniques

## Core Summary

Fine-tuning — это когда берёшь уже обученную LLM (например Llama 3) и дообучаешь её на своих данных, чтобы она лучше работала в твоей конкретной задаче. Модель уже умеет "говорить" — ты учишь её говорить *по-твоему*.

Главный вопрос в production: fine-tuning или RAG (подкидывать модели нужные документы на лету)? Часто ответ — оба вместе.

---

## Key Concepts

### 1. Full Fine-Tuning vs PEFT

**Full fine-tuning** — обновляешь ВСЕ веса модели (миллиарды параметров). Максимальная адаптация, но нужно много GPU и данных.

**PEFT (Parameter-Efficient Fine-Tuning)** — обновляешь только маленькую часть параметров (1-5%), остальное замораживаешь. Результат почти такой же, но в 10-100x дешевле.

Скрытая проблема full fine-tuning: **catastrophic forgetting** — модель настолько глубоко учит новый домен, что "забывает" базовые знания. Например, обучил на медицинских текстах — она перестала нормально писать код. PEFT этого частично избегает, потому что основные веса не трогаются.

---

### 2. LoRA & QLoRA

**LoRA (Low-Rank Adaptation)** — самый популярный метод PEFT. Идея простая:

Вместо того чтобы менять огромную матрицу весов W (допустим 4096x4096 = 16 млн чисел), ты добавляешь рядом две маленькие матрицы A и B. Их произведение AxB — это "поправка" к оригинальным весам.

**Rank (ранг)** — это размер этих маленьких матриц. Rank 8 значит матрицы будут 4096x8 и 8x4096 — это всего ~65 тысяч чисел вместо 16 миллионов. Rank 32 — чуть больше матрицы, больше ёмкости для обучения, но и больше памяти. Типичные значения: 8-64.

- Маленький rank (8-16) = быстро, дёшево, но модель может недоучиться
- Большой rank (64-128) = больше ёмкости, но теряется смысл "efficiency" в PEFT

**QLoRA** — это LoRA поверх квантизированной (сжатой) модели. Квантизация — это когда веса модели хранятся не в 32-bit (точных числах), а в 4-bit (грубых приближениях). Модель занимает в 8x меньше памяти.

Практически: QLoRA позволяет fine-tune модель на 70 миллиардов параметров на ОДНОЙ consumer GPU (RTX 4090, 24GB памяти). Без QLoRA нужен кластер из 8 серверных GPU.

### 3. Dataset Preparation & Quality

Данные для fine-tuning бывают двух типов:

**Instruction tuning data** — пары "инструкция - правильный ответ":

> Instruction: "Переведи на английский: Привет мир"
> Response: "Hello world"

**Preference data** — тройки "вопрос + хороший ответ + плохой ответ" (для alignment, см. п.9):

> Prompt: "Как похудеть?"
> Chosen: "Сбалансированная диета и регулярные тренировки..."
> Rejected: "Просто не ешь неделю..."

Форматы данных: Alpaca, ShareGPT, ChatML — это просто разные JSON-структуры для одного и того же.

**Главное правило: качество > количество.** 1000 вычищенных, правильных примеров часто лучше 100 000 шумных. Data quality — фактор номер 1 в качестве fine-tuned модели.

---

### 4. Hyperparameter Tuning

Гиперпараметры — настройки процесса обучения (не самой модели, а *как* ты её учишь):

**Learning rate (скорость обучения)** — насколько сильно модель меняет веса за каждый шаг. Самый важный параметр.
- Слишком высокий — модель "прыгает" и не сходится (diverges)
- Слишком низкий — учится вечность
- Типичные значения: 1e-4 - 2e-4 для LoRA, 1e-5 - 5e-5 для full fine-tuning (1e-4 = 0.0001)

**Epochs** — сколько раз модель проходит через весь датасет. Для instruction tuning обычно 1-3. Больше = overfitting (модель заучивает примеры наизусть вместо обобщения).

**Batch size** — сколько примеров модель обрабатывает одновременно. Ограничен видеопамятью (VRAM). **Gradient accumulation** — трюк: если в GPU влезает только 4 примера, но хочешь эффект batch size 32, модель копит градиенты за 8 шагов и потом обновляет веса разом.

**Warmup** — первые 3-10% обучения learning rate плавно растёт от 0 до целевого значения. Без этого модель может "сломаться" на первых шагах.

---

### 5. Evaluation During/After Fine-Tuning

Как понять что fine-tuning работает:

**Automated metrics:**
- **Perplexity** — насколько модель "удивляется" тексту. Ниже = лучше. Если perplexity падает на eval set — модель учится.
- **BLEU/ROUGE** — сравнивает сгенерированный текст с эталоном (overlap слов/фраз). Используется для перевода, summarization.

**LLM-as-judge** — берёшь сильную модель (GPT-4, Claude) и просишь её оценить ответы твоей fine-tuned модели. Быстрее и дешевле human eval, но у модели-судьи свои предвзятости.

**Human eval** — люди оценивают ответы. Gold standard, но дорого.

**Критически важно:** тестировать не только целевую задачу, но и что модель не деградировала в общих способностях (regression testing).

---

### 6. Merging & Model Soups

**Model merging** — объединение весов нескольких fine-tuned моделей в одну БЕЗ дополнительного обучения. Буквально: берёшь веса модели A и модели B и математически комбинируешь.

Зачем: модель A обучена на код, модель B на reasoning — merge может дать обе способности.

Методы merge:
- **SLERP** — сферическая интерполяция (плавное смешивание двух моделей)
- **TIES** — умный merge: выбирает из каждой модели только значимые изменения
- **DARE** — случайно зануляет часть параметров перед merge, снижает конфликты

**Model soups** — усреднение нескольких чекпоинтов (промежуточных сохранений) одного обучения. Часто даёт лучшую стабильность чем любой один чекпоинт.

**Ограничение:** merge работает только между моделями с одинаковым base (два fine-tune от Llama 3 — ок, Llama + Mistral — мусор).

### 7. Cost & Infrastructure

Реальные цифры:

| Сценарий | Что нужно | Стоимость за run |
|----------|-----------|-----------------|
| QLoRA 7B модели | 1x RTX 4090 (24GB) | $5-20 |
| QLoRA 70B модели | 1x A100 40GB | $30-100 |
| Full fine-tune 7B | 4x A100 80GB | $200-500 |
| Full fine-tune 70B | 8x A100 80GB | $1000-5000 |

Где арендовать GPU: RunPod, Lambda Labs, AWS SageMaker, Google Vertex AI.

Training frameworks (что запускать):
- **Unsloth** — 2-5x ускорение через оптимизированные CUDA ядра. Самый быстрый для LoRA/QLoRA.
- **Axolotl** — конфиг через YAML файл, не нужно писать код.
- **Hugging Face TRL** — стандартная библиотека, самая гибкая.

### 8. RAG vs Fine-Tuning Decision

RAG (Retrieval-Augmented Generation) — вместо дообучения модели, ты подкидываешь ей нужные документы прямо в промпт перед генерацией ответа.

Когда что выбрать:

| Ситуация | RAG | Fine-Tuning |
|----------|-----|-------------|
| Данные часто обновляются | лучше | пришлось бы переобучать |
| Нужен специфический стиль/формат | ограничен | модель учится стилю |
| Бюджет на inference минимальный | retrieval добавляет latency | всё в модели |
| Нужны знания за пределами training data | подкидываешь актуальные доки | модель знает только то на чём обучена |
| Специализированная терминология | частично | модель выучивает жаргон |

**Часто лучший ответ — оба:** fine-tune для стиля и формата + RAG для актуальных данных.

---

### 9. Post-Training Alignment

После fine-tuning модель знает домен, но может вести себя непредсказуемо (токсичные ответы, галлюцинации, игнорирование инструкций). Alignment — обучение модели *поведению*.

**RLHF (Reinforcement Learning from Human Feedback):**
1. Люди ранжируют ответы модели (какой лучше)
2. На этих данных обучается reward model (модель-оценщик)
3. Основная модель оптимизируется через PPO (алгоритм reinforcement learning) чтобы получать высокие оценки от reward model

Дорого, нестабильно, но именно так обучены ChatGPT и Claude.

**DPO (Direct Preference Optimization):** упрощение RLHF — убирает reward model и PPO. Напрямую оптимизирует модель на парах "хороший ответ vs плохой ответ". Проще, стабильнее, стал стандартом в 2024-2025.

**ORPO (Odds Ratio Preference Optimization):** ещё проще — объединяет instruction tuning + alignment в один шаг обучения вместо двух.

## Architecture Flow

**Raw Data** (JSON, CSV, текстовые файлы)

⬇️ **Data Preparation** — Чистка, форматирование в Alpaca/ChatML, split на train/eval, фильтрация мусора

⬇️ **Base Model Selection + Quantization** — Выбор: Llama 3, Mistral, Qwen, Gemma. Размер vs качество. 4-bit если QLoRA.

⬇️ **Fine-Tuning** — LoRA/QLoRA конфиг: rank, learning rate, epochs, target layers

⬇️ **Evaluation** — Perplexity, LLM-as-judge, regression tests, human eval

⬇️ **Alignment (optional)** — DPO / ORPO на preference data, если модель user-facing

⬇️ **Merge / Export** — Merge LoRA в base model, export в GGUF (формат для Ollama/llama.cpp), model soups если несколько runs

⬇️ **Deployment** — vLLM, TGI, Triton (серверы для inference), или Ollama для локального запуска

## Important Tools / Frameworks

| Tool | Зачем нужен |
|------|-------------|
| Hugging Face TRL | Стандартная Python-библиотека для SFT (supervised fine-tuning), DPO, RLHF |

| bitsandbytes | Библиотека для 4/8-bit квантизации, нужна для QLoRA |
| PEFT (HF) | Библиотека parameter-efficient методов (LoRA, Adapters, Prefix Tuning) |
| mergekit | CLI-инструмент для model merging (TIES, DARE, SLERP) |
| W&B / MLflow | Experiment tracking — графики loss, сравнение runs, сохранение артефактов |
| vLLM | Сервер для inference fine-tuned моделей (быстрый, с batching) |

## Examples

**Пример 1 — Медицинский ассистент:**
- Base: Llama 3 8B, QLoRA rank 32 на 5000 медицинских Q&A
- Eval: вопросы из медицинских экзаменов (accuracy) + общие бенчмарки (не деградировала ли модель)
- Alignment: DPO на парах "безопасный медицинский совет vs опасный"
- Deploy: vLLM за FastAPI + defensive system prompt

**Пример 2 — Code assistant для внутренней кодовой базы:**
- Base: CodeLlama 13B, LoRA на internal codebase (функции, docstrings, PR comments)
- RAG поверх: актуальная документация и Jira тикеты (данные меняются — RAG лучше)
- Eval: проходят ли сгенерированные тесты + human review
- Merge: code-fine-tune + instruction-fine-tune через TIES

## Why This Matters

Fine-tuning — мост между general-purpose LLM и реальным продуктом. Prompt engineering имеет потолок: когда нужен специфический формат, терминология, стиль или поведение на уровне модели — fine-tuning единственный путь.

С QLoRA порог входа упал до $10 и одной потребительской GPU — это сделало кастомизацию LLM доступной всем.

Решение "RAG vs fine-tuning vs оба" — одно из ключевых архитектурных решений в любом LLM проекте, и его спрашивают на каждом интервью.

## Glossary

| Term | Что это |
|------|---------|
| PEFT | Parameter-Efficient Fine-Tuning — методы дообучения малой части параметров |
| LoRA | Low-Rank Adaptation — добавление маленьких обучаемых матриц рядом с основными |
| Rank | Размер добавленных LoRA-матриц. Больше = больше ёмкости, но дороже |
| Quantization | Сжатие весов модели (32-bit в 4-bit) для экономии памяти |
| VRAM | Видеопамять GPU — главное ограничение при fine-tuning |
| Catastrophic forgetting | Модель забывает старые знания при обучении на новых данных |
| Overfitting | Модель заучивает примеры наизусть вместо обобщения |
| Divergence | Модель "взрывается" — loss растёт вместо падения (обычно из-за высокого LR) |
| Instruction tuning | Fine-tuning на парах "инструкция - ответ" |
| Preference data | Тройки "вопрос + хороший ответ + плохой ответ" для alignment |
| DPO | Direct Preference Optimization — alignment без reward model |
| RLHF | Reinforcement Learning from Human Feedback — alignment через reward model + RL |
| Model merging | Математическое объединение весов нескольких моделей без обучения |
| GGUF | Формат файлов для запуска моделей через llama.cpp / Ollama |
| Gradient accumulation | Накопление градиентов за несколько шагов для имитации большого batch size |
| Regression testing | Проверка что модель не стала хуже в том что раньше умела |
